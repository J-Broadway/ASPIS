`aspis.clause.schema` defines the machine-readable markdown meta-header used to register and resolve addressable governance clauses.

Clause identity is effective identity, not just authored identity. `kind` classifies a clause but does not determine clause identity, and optional discovery metadata must not affect identity semantics.

Heading-derived identity is inferable only for schema-style headings with exactly one canonical schema token (for example, ``### `aspis.clause.schema```).

When heading identity is inferable, explicit `id` is optional; if explicit `id` is present it must exactly match the inferred identity after normalization.

When heading identity is not inferable, explicit `id` is required.

Any ambiguous heading-derived identity, invalid explicit/inferred identity grammar, or explicit/inferred mismatch fails closed.

`owner_doc` may be a literal canonical doc identity or an approved template token. `owner_doc: {doc_id}` is valid inline clause authoring and must resolve deterministically to the enclosing governed document's canonical `doc_id` during extraction/indexing.

Unresolved/ambiguous templated `owner_doc` values, or `owner_doc` values that do not map to exactly one governed canonical document, are blocking.

`blocking_behavior` defines the semantic deterministic contract outcome for a known condition and the enforcement behavior that follows from that outcome.

`outcome` answers whether the condition blocks progress. `enforcement` answers how the system should respond when the condition is encountered.

This distinction is intentional:

- `blocking` or `non_blocking` is the contract decision

`blocking_behavior` must not be defined by advisory prose alone. If blocking enforcement emits or remaps a reason code, that code must be registered by `aspis.registry.reason-code-bindings.schema`.

Tooling route for ASPIS runtime contracts.

`aspis.tools` defines the operational runtime layer that executes ASPIS governance behavior.

Tool contracts are execution surfaces, not replacement authority surfaces:
- Protocol and instance governance documents remain the canonical source of truth.
- Tools consume those sources to parse, validate, route, and generate deterministic artifacts.
- Generated outputs (for example workspace registries) are derived evidence surfaces and must remain traceable to governing clauses.

In practice, `aspis.tools` is the bridge between:
- specification contracts (`aspis.policy.*`, `aspis.clause.*`, `aspis.registry.*`)
- and enforceable runtime behavior (`lint`, workspace discovery, clause parsing/resolution)

Agents should treat `aspis.tools.*` clauses as operational guidance for how conformance is evaluated and how introspective navigation is performed at runtime.

Route to ASPIS lint semantics and lint-facing contracts.

Lint execution context:

1) Workspace discovery
- Read `aspis.yaml` workspace manifest.
- Resolve protocol authority surface from `workspace.protocol_root`.
- Resolve target namespace from `aspis lint <doc_domain>` (or explicit `--target`) and select exactly one owning instance from `instances[]` (`namespace`, `root`, `governance_doc`).
- Determine authority target using `aspis.authority.selection.schema` before registration traversal.

2) Registration pass
- Traverse markdown governance docs in protocol authority + the single selected instance authority.
- Parse canonical clause headers (`<!-- aspis:clause -->`) and optional registry slots (`<!-- aspis:registry-slots -->`).
- Build an effective registry from discovered protocol + instance clauses.

3) Validation pass
- Enforce lineage and namespace constraints across surfaces.
- Validate owner-doc resolution and duplicate/ambiguous identities.
- Validate in-scope domain references in clause sections against the effective registry.
- When remediation is required, route through authority selection before deeper registration-schema traversal.

4) Output and artifacts
- Emit deterministic lint findings and summary payload for automation.
- Write generated selected-instance registry artifact to `<instance-root>/.aspis/aspis.registry.yaml`.

Scope boundary:
- Lint semantic reference validation applies to clause sections.
- Top-level descriptive prose outside clause sections (for example Purpose statements) is not scanned for unresolved domain references.
